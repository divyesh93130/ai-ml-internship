# AI-Powered Food Nutrition Analyzer — Prototype

This is a working Phase-1 MVP skeleton for the internship project. It covers:
- Fuzzy food-name matching (typo/casual-text tolerant)
- Unit → gram conversion
- Per-100g → requested-quantity nutrient scaling
- A minimal API + frontend to demo the full flow

## How to run it

**1. Backend**
```bash
cd backend
pip install -r requirements.txt
uvicorn main:app --reload
```
This starts the API at `http://127.0.0.1:5500`. Visit `http://127.0.0.1:5500/docs`
for an interactive test console (auto-generated by FastAPI).

**2. Frontend**
The app should be opened from the backend URL, not from VS Code Live Server or `file://`.
Run `start.bat` from the repo root and then open the browser at
`http://127.0.0.1:5500`.

If you prefer to start the backend manually, do so first on
`http://127.0.0.1:5500`, then open the browser to that URL.

Do not open `frontend/index.html` directly in Live Server or via `file://`.
The backend must be running before clicking Analyze.

```bash
cd backend
.venv\Scripts\python main.py
```

## Step-by-step: how a request flows through the system

1. User types food name + quantity + unit in the frontend form.
2. Frontend POSTs `{ food_name, quantity, unit }` to `/analyze`.
3. **Matching engine** (`matching.py`) runs fuzzy matching (`rapidfuzz`) against
   all food names in the database, returns the best match + confidence score,
   plus 2 runner-up matches (useful for a "did you mean...?" UI later).
4. **Unit converter** (`unit_conversion.py`) looks up the unit in
   `unit_conversion.csv` and converts quantity → grams.
5. Nutrient values (stored per-100g in `nutrition_data.csv`) are scaled by
   `grams / 100`.
6. API returns matched food, confidence, and scaled nutrients as JSON.
7. Frontend renders the nutrient cards.

## What each sub-team should extend from here

- **Data & Database team**: replace `nutrition_data.csv` with a merged
  USDA + IFCT 2017 dataset (Section 8 of the brief), move it into SQLite/Postgres,
  and expand `unit_conversion.csv` with per-food overrides (e.g. "1 piece roti"
  should differ from "1 piece samosa" — right now "piece" is a flat 40g fallback).
- **AI/Matching team**: swap `FoodMatcher` for the commented-out
  `SemanticFoodMatcher` (sentence-transformers) once fuzzy matching starts
  missing synonyms (e.g. "chana" vs "chickpeas"). Also build the Section 4.3
  "unknown food" estimator — when no match clears the confidence threshold,
  find nearest foods by category instead of just returning a 404.
- **Backend/API team**: add multi-item parsing ("2 rotis and 1 bowl dal"),
  request logging, and input validation hardening.
- **Frontend/QA team**: add a "did you mean...?" prompt using the
  `alternative_matches` field already returned by the API, and test edge cases
  (typos, unknown units, zero/negative quantities, unknown foods).

## Known limitations (Phase 1)

- Only 30 sample foods — needs a real merged dataset.
- "piece" and "slice" units are flat fallbacks, not food-specific.
- No handling yet for multiple foods in one sentence.
- No image-based food recognition (Phase 2 stretch goal).
